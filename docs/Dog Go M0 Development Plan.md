# Dog Go M0 开发计划

**计划版本：** 1.0  
**对应 PRD：** Dog Go v2.2 原生 MVP 开发验证版  
**目标平台：** iOS 17+  
**技术栈：** SwiftUI、SwiftData、XCTest  
**预计周期：** 10 个工作日  
**计划状态：** Day 10 已完成，M0 内部演示候选（真机验收待执行）

---

## 1. 阶段目标

M0 的目标不是制作完整宠物应用，而是交付一个可以在真实设备运行的内部可玩版本，验证以下核心闭环是否成立：

```text
领养狗狗
  -> 离开一段时间
  -> 狗狗独立经历事件
  -> 用户回来发现环境与生活片段
  -> 用户作出轻量回应
  -> 回应形成记忆
  -> 后续事件引用这段记忆
```

完成 M0 后，应能判断：

1. 狗狗的离线生活是否具有连续性，而非随机动画感。
2. 用户回应是否能以可理解的方式影响后续事件。
3. 狗狗偶尔不采纳建议时，是否仍显得有性格且不冒犯用户。
4. 当前数据模型和事件格式是否足以扩展到 M1 的三只狗和 27 个事件。

## 2. M0 开发范围

### 2.1 包含内容

- 1 只柴犬，默认名字“栗子”，允许用户改名。
- 2 个固定性格特质：好奇、爱整洁。
- 1 个住所场景。
- 3 种当前行为：睡觉、观察、玩耍。
- 9 个普通生活事件。
- 3 个连续事件。
- 3 个首次体验保底事件。
- 领养、首页、生活片段、回应、记忆回顾完整闭环。
- 在线陪伴循环：叫名字、轻轻摸摸、安静陪伴。
- 停留期间的自主行为变化与即时性格反应。
- SwiftData 本地存储。
- 离线事件生成与带种子的加权选择。
- Debug 构建中的时间推进和状态检查工具。
- 核心事件引擎单元测试。
- 基础 VoiceOver 标签和减少动态效果支持。

### 2.2 明确不包含

- 柯基、金毛及多犬选择。
- 街角外景和外出动画。
- 推送通知。
- 分享卡片。
- 云端服务、账号、同步和生成式 AI。
- 商店、付费、装扮、货币和签到。
- 正式分析 SDK；M0 只保留本地调试日志接口。
- 完整美术量产和连续背景音乐。

## 3. 用户流程与页面

### 3.1 首次启动

1. 产品承诺页。
2. 柴犬介绍页，展示具体性格行为。
3. 改名与确认领养。
4. 初遇事件。
5. 首页，并提示用户可以先去忙自己的事。

**验收要求：** 首次领养在 90 秒内完成，流程不超过 5 个主要界面状态。

### 3.2 回访流程

1. 首页展示狗狗当前行为。
2. 显示“你不在时”新增片段数量。
3. 用户打开片段列表并逐条查看。
4. 重要片段提供 2-3 个回应和“安静陪它一会儿”。
5. 选择后立即显示狗狗反应。
6. 重要事件与回应写入“我们的日子”。

### 3.3 M0 页面清单

| 页面 | 核心职责 | M0 状态 |
|---|---|---|
| 产品承诺页 | 解释狗狗会独立生活且不会惩罚用户 | 开发 |
| 领养页 | 展示柴犬性格、修改名字、确认领养 | 开发 |
| 初遇页 | 触发首个保底事件 | 开发 |
| 回访首页 | 展示场景、行为、新片段数量和轻互动入口 | 开发 |
| 生活片段页 | 展示事件事实、情绪、痕迹和回应 | 开发 |
| 我们的日子 | 按时间展示重要事件与用户回应 | 开发精简版 |
| Debug 面板 | 推进时间、查看状态、强制重新载入 | 仅调试构建 |

### 3.4 在线陪伴循环

离线事件负责制造回访理由，在线陪伴负责建立关系。用户在首页停留时，狗狗不会静止等待指令，而会在当前行为中自然呼吸、眨眼、转耳、观察环境，并偶尔切换姿态。

M0 提供三种无奖励、无任务压力的轻互动：

- **叫名字：** 栗子可能回头、靠近，或只转动耳朵继续忙自己的事。
- **轻轻摸摸：** 根据心境接受触摸、短暂回应，或自然避开后继续当前行为。
- **安静陪伴：** 进入 20-30 秒低干扰状态，只保留呼吸、窗帘、光影和环境音。

用户单次在线体验目标为 1-3 分钟。互动不产生货币、进度奖励或永久负面后果，但重要回应可写入关系记忆。

## 4. 技术设计

### 4.1 工程结构

```text
DogGo/
  App/
  Features/
    Adoption/
    Home/
    LifeMoments/
    Memories/
    DebugTools/
  Domain/
    Models/
    LifeEngine/
    Repositories/
  Data/
    Persistence/
    EventCatalog/
  DesignSystem/
  Resources/
    EventContent/
    Assets/
DogGoTests/
```

采用轻量单向数据流：视图发送用户动作，Feature Store 调用领域服务并更新可观察状态。事件选择、状态变化和持久化逻辑不得写在 SwiftUI View 内。

### 4.2 SwiftData 实体

#### DogProfile

- `id`
- `name`
- `breed`
- `adoptedAt`
- `traitWeights`

#### DogState

- `mood`
- `energy`
- `socialTendency`
- `curiosity`
- `currentBehavior`
- `lastSimulatedAt`

#### LifeEventRecord

- `id`
- `definitionID`
- `occurredAt`
- `sceneID`
- `factPayload`
- `emotion`
- `visualTraceID`
- `isViewed`
- `selectedResponseID`

#### MemoryRecord

- `id`
- `createdAt`
- `sourceEventID`
- `responseID`
- `tags`
- `referencedByEventIDs`

#### Relationship

- `stage`
- `milestoneIDs`
- `updatedAt`

M0 不单独创建用户实体，不保存性别、位置或自由输入内容。狗狗名字是唯一自由输入字段，只保存在本地。

### 4.3 事件定义格式

事件内容使用项目内 JSON 配置，事实与展示文案分离。每个事件至少包含：

```json
{
  "id": "home_window_bird_01",
  "category": "ordinary",
  "timeWindows": ["daytime"],
  "baseWeight": 10,
  "preferredTraits": ["curious"],
  "excludedRecentEventIDs": [],
  "requiredMemoryTags": [],
  "stateEffects": {},
  "memoryOutputTags": ["noticed_bird"],
  "followUpEventIDs": ["home_window_bird_02"],
  "sceneID": "home",
  "visualTraceID": "feather_near_window",
  "textVariants": [],
  "responses": []
}
```

JSON 在应用启动时解码为不可变定义；已发生事件将事实快照写入 SwiftData，后续更新文案配置不能改变历史事实。

### 4.4 离线生活引擎

每次应用进入前台时：

1. 读取 `lastSimulatedAt` 与当前时间。
2. 判断是否跨过事件触发阈值或时间窗口。
3. 最多生成 3 个离线事件，超过 24 小时也不累积惩罚。
4. 过滤不满足前置条件或近期重复的事件。
5. 按性格、时间、记忆、连续故事和重复惩罚计算权重。
6. 使用稳定种子选择事件，并写入事实快照。
7. 应用状态影响，更新当前行为和 `lastSimulatedAt`。

幂等键由狗狗 ID、模拟时间窗口和事件序号组成。同一窗口重复启动不得生成重复记录。

### 4.5 时间与随机策略

- 生产逻辑通过 `Clock` 协议读取时间，测试中注入固定时钟。
- 随机选择通过 `SeededRandomSource` 注入，不直接调用全局随机函数。
- Debug 时间推进不修改系统时间，只调整测试时钟偏移量。
- 系统时间倒退时不删除历史、不逆向模拟；等待时间重新超过最后模拟时间。
- 单次回访最多生成 3 条事件，防止修改时间产生无限内容。

## 5. M0 内容设计

### 5.1 首次体验保底事件

| ID | 事件 | 目的 |
|---|---|---|
| `first_meeting` | 栗子在新家观察用户，最后靠近半步 | 建立初识与谨慎的主动性 |
| `first_short_leave` | 用户离开后，栗子检查门口并回到垫子上 | 解释离开期间仍有生活 |
| `first_memory_reference` | 栗子以行动回应用户上次的态度 | 证明回应能留下影响 |

### 5.2 普通事件主题

1. 窗边观察小鸟。
2. 把玩具藏到垫子下面。
3. 发现阳光移动的位置。
4. 听见楼道脚步声。
5. 整理被弄歪的小毯子。
6. 对纸袋产生兴趣。
7. 在水碗旁发现自己的倒影。
8. 睡醒后寻找更舒服的位置。
9. 把掉落的小物件推到墙边。

### 5.3 连续事件链

#### 窗边来客

`注意小鸟 -> 用户回应 -> 再次等待小鸟 -> 留下一根羽毛痕迹`

#### 消失的玩具

`藏起玩具 -> 用户回应 -> 栗子决定是否透露位置 -> 玩具重新出现`

#### 歪掉的小毯子

`发现毯子歪了 -> 用户建议 -> 栗子自行整理或保留新摆法 -> 后续在该位置休息`

三条事件链在 M0 中各实现一个前序事件和一个后续事件，其余节点可作为 M1 扩展内容。

## 7. 测试计划

### 7.1 单元测试

- 相同种子与状态产生相同事件结果。
- 不满足时间、性格或记忆条件的事件不会入选。
- 最近发生的事件受到重复惩罚或被排除。
- 连续事件在满足记忆标签后提高权重。
- 同一模拟窗口重复执行不会重复写入。
- 超过 24 小时离开不产生负面状态或超过 3 个事件。
- 系统时间倒退不会删除、重复或篡改历史事件。
- 用户回应正确写入记忆并可被后续事件引用。

### 7.2 集成与手工测试

- 首次安装到完成领养。
- 杀掉应用后重新启动，领养和事件仍存在。
- 推进 10 分钟、数小时和一天后的回访表现。
- 连续打开三次不展示完全相同的事件组合。
- 无事件、事件配置损坏和持久化失败时的降级界面。
- VoiceOver 阅读顺序、动态字体和减少动态效果。
- iPhone 小屏与常用尺寸的布局。

## 8. M0 完成定义

同时满足以下条件才视为 M0 完成：

- 可以在 iOS 17+ 真机完成完整核心流程。
- 首次领养不超过 90 秒。
- 离开达到条件后生成 1-3 个事件。
- 相同离线窗口不会生成重复事实。
- 用户回应在 500 毫秒内得到即时反馈。
- 至少一个后续事件能够明确引用前序事件或用户回应。
- 连续三次回访不出现完全相同的事件组合。
- 事件事实生成后不会因重新载入文案而改变。
- 用户离开超过 24 小时不产生惩罚、损失或求救文案。
- 核心单元测试全部通过，无阻塞级崩溃。
- Debug 工具不进入正式 Release 构建。

## 9. 主要风险与控制

| 风险 | 控制方式 |
|---|---|
| 美术未准备导致页面无法推进 | M0 使用原创基础占位插画与简单分层动画，接口保持可替换 |

| Debug 时间推进污染正式逻辑 | 统一使用可注入 Clock，并用编译条件隔离入口 |
| 两周内内容写作拖慢开发 | Day 2 前冻结 15 个事件的事实、条件和回应，文案可后续润色 |

## 11. 需要审核确认的决策

1. **默认狗狗：** 已确认使用柴犬，默认名“栗子”。
2. **视觉策略：** 已确认采用更温暖的“静谧窗景”城市绘本方向；正式场景使用可动画分层资产。
3. **首页表现：** 已确认使用 2D 分层场景，以呼吸、眨眼、转耳、回头和姿态切换等克制动作表现生命感。

5. **开发节奏：** 按 10 个工作日完成 M0，Day 5 安排一次中期审核。
6. **设备方向：** M0 仅支持 iPhone 竖屏，不支持 iPad 和横屏。

# M0.5「生命感可视化」开发计划

**阶段状态：** 已审核，Day 14 已完成

**预计周期：** 10 个工作日（Day 11–Day 20）

**核心目标：** 将 M0 已完成的事件、记忆和陪伴系统转化为用户可以直接看见和感受到的狗狗生活。

## M0.5 阶段目标

完成一条正式视觉闭环：

```text
打开 App
  -> 看见会动的柴犬和房间
  -> 离开后发生事件
  -> 回来观看事件画面
  -> 回应狗狗
  -> 场景与狗狗产生变化
  -> 记忆在后续生活中被再次引用
```

本阶段优先解决 PRD 第 19 项反馈：缺少正式狗狗形象、缺少动态表现、生活片段以文字为主，以及没有可感知的生活场景。

## M0.5 开发范围

### 包含内容

- 正式柴犬角色与统一的“静谧窗景”城市绘本画风。
- 温暖室内场景，以及窗户、窗帘、家具、地面、光影等分层资产。
- 狗狗分层动画播放器与状态驱动接口。
- 呼吸、眨眼、转耳、摇尾、回头、趴下、起身等基础动作。
- 首页在线待机、呼唤、轻摸和安静陪伴反馈。
- 至少一个完整离线事件的画面回放。
- 昼夜变化、玩具位移、纸袋、光斑和窗帘等环境痕迹。
- 生活片段与“我们的日子”的画面优先改造。
- 减少动态效果适配、性能检查和真机视觉验收。

### 暂不包含

- 新犬种和多犬选择。
- 新住所与室外场景。
- 商店、付费、装扮和货币系统。
- 大规模新增事件内容。
- 云端 AI、账号和同步。
- 复杂自由移动、骨骼物理和 3D 表现。

## 动画规格

首批动作采用克制、自然的表现方式：

- **呼吸：** 持续但轻微的待机循环。
- **眨眼：** 使用随机间隔，避免机械重复。
- **转耳：** 响应环境声或用户呼唤。
- **摇尾：** 用户出现、互动或积极回应后触发。
- **回头：** 从独处状态转向用户。
- **趴下与起身：** 用于睡眠、观察和玩耍状态切换。
- **玩玩具：** 用于离线事件和活跃状态。
- **入睡：** 用于夜间或低精力状态。

动画不制作成固定循环视频，而是由当前行为、事件、时间和用户操作共同驱动。

## 首个离线事件视觉闭环

首个完整事件暂定为“第一次短暂分别”：

1. 用户离开后，柴犬短暂看向门口。
2. 柴犬回到窗边或垫子上，继续自己的生活。
3. 窗帘、光线和环境痕迹随时间缓慢变化。
4. 用户回来时，柴犬抬头、转耳并轻轻摇尾。
5. 用户回应后，狗狗立即产生可见动作反馈。
6. 回应写入现有记忆系统，并可被后续事件引用。

## M0.5 验收标准

- 首页第一眼能够明确认出正式柴犬，而不是通用图标或占位图。
- 狗狗待机 30 秒内至少自然出现两种动作。
- 动作没有持续抖动、机械循环或明显干扰阅读的问题。
- 用户主要通过画面理解离线事件，文字只负责补充事实与情绪。
- 场景能够表现昼夜变化和至少两种事件痕迹。
- 用户回应后 500 毫秒内出现可见反应。
- 开启“减少动态效果”后提供合理的静态或弱动画替代。
- 常用 iPhone 尺寸运行流畅，无明显掉帧和异常耗电。
- 首次体验用户能够准确说出“狗狗刚才做了什么”和“它是什么性格”。
- 完成一次真机安装、冷启动、持久化和完整视觉闭环回归。
