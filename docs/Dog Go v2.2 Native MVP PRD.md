# Dog Go v2.2 原生 MVP 开发验证版产品需求文档

**版本：** 2.2  
**文档状态：** 已批准进入原生 MVP 开发，通过 TestFlight 验证核心假设  
**首发平台：** iOS 17+  
**产品类型：** 自主生活型数字宠物陪伴应用  
**核心决策：** 先验证用户是否会惦记一只有自己生活的数字狗，再扩充犬种、场景和养成系统

---

## 1. 执行摘要

Dog Go 面向在忙碌、重复和信息过载的生活中，希望获得短暂安静陪伴的城市成年人。用户领养一只有独特外观和性格的数字狗；当用户离开应用时，狗狗仍会按照自己的性格、节奏、记忆和近期经历继续生活。用户再次打开应用，不是为了完成任务或避免惩罚，而是为了发现狗狗刚刚经历了什么、回应它，并逐渐形成只属于双方的共同记忆。

Dog Go 的首要产品假设是：**一只拥有连续生活轨迹、有限自主性和关系记忆的数字狗，能够让用户产生“我想回来看看它”的非功利性回访动机。** v2.2 不追求完整养宠玩法，而是用最小范围验证这种惦记感是否真实存在。

### 一句话定位

> Dog Go 是一只有自己生活的数字狗。你不在的时候，它也在认真生活。

### 核心体验承诺

- 不要求用户打卡或完成现实任务。
- 不因用户离开而惩罚、责备或制造内疚。
- 每次回来都有机会发现一个新的生活片段。
- 狗狗会受到用户回应影响，但不会完全服从用户。
- 关系通过共同经历形成，而不是通过重复点击堆数值。

---

## 2. 问题陈述

### 2.1 目标用户面临的问题

部分城市成年人每天在工作、通勤、家庭事务和碎片化内容之间切换。短视频和信息流能快速刺激注意力，却不一定带来真正的放松；传统效率工具又要求用户继续规划、记录和完成目标。用户缺少一种无需表现、无需社交、无需持续投入，也能提供轻微期待和稳定陪伴的数字体验。

### 2.2 用户痛苦

- 想短暂放松，却习惯性进入重复的信息流。
- 想获得陪伴，但不想承担真人社交的即时回应压力。
- 喜欢宠物，却可能受居住、时间、费用或家庭条件限制。
- 传统电子宠物通过饥饿、生病和连续签到维持活跃，容易产生负担。
- AI 陪聊需要用户持续发起话题，长期使用可能重新变成一种对话任务。

### 2.3 当前证据状态

以上问题来自产品假设和市场观察，尚未通过 Dog Go 目标用户访谈验证。项目不再设置独立访谈和轻量概念原型作为开发前置条件，而是直接通过原生 MVP 收集证据。

因此接受以下约束：

1. 用户认知、三日回访和付费意愿统一在 TestFlight 测试中验证。
2. 因跳过前期发现阶段而增加的方向风险，由严格的 MVP 范围和阶段性停止条件控制。
3. 第一版保持本地优先、内容可配置和可快速调整，不提前扩充美术、犬种或商业系统。
4. 原生 MVP 完成后仍须招募 15-20 名目标用户完成三天封闭测试和结束访谈。
5. 至少进行一次真实付费或订金测试，不以问卷中的口头意愿代替。

### 2.4 待验证的问题陈述

> 当城市成年人被重复生活和碎片内容消耗时，他们需要一个无需完成任务、无需维持对话的安静数字空间，通过观察一只拥有独立生活轨迹的狗狗，获得几分钟的放松、期待和陪伴感。

## 3. 目标用户与使用任务

### 3.1 首要用户

**独居或经常独处的城市上班族，25-38 岁。**

- 日常节奏相对固定，工作日重复感明显。
- 高频使用手机，但对短视频和社交媒体产生疲惫。
- 喜欢宠物或治愈型内容。
- 每天愿意进行 1-3 次、每次 1-3 分钟的轻互动。
- 不希望应用要求自律、记录情绪或持续聊天。

选择该群体作为首要用户，是为了减少 MVP 的生活模式和内容分支。自由职业者、自律型用户和中年家庭用户属于后续验证人群，不进入首轮范围。

### 3.2 Jobs to Be Done

**主要任务**

> 当我在通勤、工作间隙或回家后感到疲惫时，我想进入一个安静的小世界，看看我的狗狗正在做什么，从而短暂离开信息流并感受到陪伴。

**情感任务**

> 我想感觉某个小生命拥有与我相关、但不完全依附于我的生活。

**社会任务**

> 我希望偶尔分享一个有趣或温暖的狗狗生活片段，但不必加入社区或维持社交关系。

### 3.3 非目标用户

- 寻找高强度宠物养成、战斗或收集玩法的玩家。
- 需要任务管理、习惯追踪或运动激励的用户。
- 主要期待无限 AI 聊天的用户。
- 希望精确模拟真实犬只护理的用户。

## 4. 战略与竞品背景

### 4.1 竞品类别

| 类别 | 代表产品 | 核心机制 | Dog Go 的预期差异 |
|---|---|---|---|
| 自我照顾宠物 | Finch | 完成目标驱动宠物成长 | 不设置现实任务，不要求改善自己 |
| 专注陪伴 | Focus Friend | 专注计时产生宠物奖励 | 离开应用不是任务，狗狗会独立生活 |
| AI 宠物 | Stride、Mochi Crew | 聊天、记忆、步数和宠物人格 | 不以聊天为主，重点是可发现的生活轨迹 |
| 桌面共同养宠 | Widgetable | 喂养、组件、好友共同照顾 | 单人、低负担、无强制照料 |
| 放置收集 | Neko Atsume | 离线等待宠物来访 | 同一只狗拥有连续性格、事件和关系记忆 |

### 4.2 差异化假设

Dog Go 不应依靠“犬种更多”“AI会聊天”或“状态更丰富”竞争。它唯一值得验证的差异是：

> 用户每次回来，都能通过环境痕迹、事件记录和狗狗反应，发现它在自己离开期间经历了一段连续、可理解、会影响未来的生活。

### 4.3 反定位

Dog Go 不是：

- 套着宠物外观的打卡工具。
- 需要用户维持生命值的电子鸡。
- 随时等待用户输入的聊天机器人。
- 由随机动画组成的宠物屏保。
- 模仿某个现有游戏视觉资产的换皮产品。

## 5. 产品原则

1. **回来靠好奇，不靠内疚。** 用户离开期间不得产生不可逆损失。
2. **狗狗有自己的生活。** 用户能影响，但不能完全控制狗狗。
3. **连续性高于内容数量。** 一个被记住的事件胜过十个随机动画。
4. **轻互动，不做任务。** 单次体验控制在 1-3 分钟，不要求完成清单。
5. **温柔但不谄媚。** 狗狗可以忙、困、闹别扭或有自己的选择，但不责备用户。
6. **事实由规则决定，文案负责表达。** 如使用生成式 AI，AI 不得改变事件事实和关系状态。
7. **原创建立识别。** 视觉语言只参考治愈、低刺激和温暖感，不复制现有游戏角色、场景或界面。

## 6. 核心产品循环

```text
用户离开
  -> 狗狗根据时间、性格、状态和记忆经历事件
  -> 用户回来发现当前场景及生活痕迹
  -> 用户查看一段简短事件
  -> 用户选择回应、轻互动或安静陪伴
  -> 狗狗形成反应和一条关系记忆
  -> 记忆影响未来事件权重
  -> 用户离开，狗狗继续生活
```

### 6.1 首次价值时刻

用户完成领养后立即看到一段短暂的初遇事件，并获知：

> “你可以去忙自己的事。下次回来时，它可能已经有新故事了。”

首次离开 10-30 分钟后，应用准备一个确定可触发的回访事件，确保用户在第一天理解“离开也会产生内容”。

### 6.2 每次打开的体验结构

1. **看见当前状态：** 狗狗正在进行一个具体行为。
2. **发现变化：** 场景中出现一个与近期事件有关的视觉痕迹。
3. **阅读片段：** 1-3 句话说明刚刚发生的事情。
4. **选择回应：** 提供 2-3 个态度不同但无对错的回应，或选择不打扰。
5. **留下记忆：** 重要片段进入“我们的日子”。

## 7. MVP范围

### 7.1 MVP目标

在 4-6 周内交付可通过 TestFlight 分发的原生 iOS MVP，验证用户是否会因为“想知道狗狗经历了什么”而在三天内主动回来，而不是验证犬种收集、装扮付费或完整养宠系统。

### 7.2 开发里程碑范围

#### M0：内部可玩版本

- 1 只狗，默认使用柴犬。
- 1 个住所场景。
- 9 个普通事件、3 个连续事件、3 个首次体验保底事件。
- 睡觉、观察、玩耍 3 种当前行为。
- 完成领养、回访首页、生活片段、回应和基础记忆闭环。
- 使用 SwiftData 本地持久化。
- 提供仅限 Debug/TestFlight 构建的时间推进和状态检查工具。

#### M1：TestFlight 验证版本

- 扩展为 3 只候选狗和 2 个场景。
- 补齐 18 个普通事件、6 个连续事件和外出行为。
- 完成“我们的日子”、温柔通知、分享卡片和最小分析埋点。
- 达到本章下述全部 P0 功能与验收标准。

### 7.3 P0功能

#### A. 精简领养流程

- 一页产品承诺说明。
- 从 3 只候选狗狗中领养 1 只，而非无限随机。
- 3 个犬种：柴犬、柯基、金毛。
- 每只候选狗包含固定外观、名字建议和 2 个初始性格特质。
- 用户可以改名。
- 不询问用户性别。
- MVP 统一采用“规律城市生活”初始节奏。

**验收标准**

- 新用户可在 90 秒内完成领养。
- 用户确认前能清楚了解狗狗不可随意替换。
- 删除应用或存档损坏不会导致付费资产损失；MVP 不售卖资产。

#### B. 狗狗生活引擎

生活引擎由四层组成：

1. **节奏层：** 早晨、白天、傍晚、夜间四个时间窗口。
2. **性格层：** 好奇、谨慎、黏人、独立、贪吃、爱整洁等权重。
3. **记忆层：** 保存最近 10 条重要事件和用户回应。
4. **事件层：** 从满足条件的事件中进行加权选择，并写入不可修改的事件事实。

MVP至少包含：

- 18 个普通生活事件。
- 6 个由前序选择触发的连续事件。
- 3 个首次体验保底事件。
- 4 种当前行为：睡觉、观察、玩耍、外出。
- 每只狗至少有 2 条专属事件倾向。

**验收标准**

- 同一用户连续三次打开时，不得展示完全相同的事件组合。
- 至少 30% 的第二天事件能够引用前一天的事件或用户回应。
- 事件事实一旦生成，不因文案刷新而改变。
- 狗狗可以选择与用户建议不同的行动，并给出符合性格的可理解原因。

#### C. 回访首页

首页只承担三件事：

- 展示狗狗当前所在场景和动作。
- 显示“你不在时”新增片段数量。
- 提供“看看发生了什么”和一次轻互动入口。

不显示饥饿、口渴、精力进度条，不显示连续签到。

#### D. 生活片段与回应

- 每个片段包括时间、事件、狗狗情绪和一个视觉痕迹。
- 用户可从 2-3 个回应中选择一个，或选择“安静陪它一会儿”。
- 回应不设正确答案，不奖励货币。
- 重要回应写入关系记忆，并影响后续事件权重。

#### E. 我们的日子

- 按时间展示共同经历。
- 自动收藏初遇、第一次外出、第一次引用用户回应等里程碑。
- 支持把单个片段导出为无个人敏感信息的分享卡片。

#### F. 温柔通知

- 默认每天最多 1 条，可完全关闭。
- 通知只报告狗狗正在经历的事情，不使用“它需要你”“它很伤心”等压力文案。
- 示例：“它今天第一次自己去了街角。回来时鞋上有一点泥。”

### 7.4 P1功能

仅在 TestFlight 验证通过后进入开发：

- 增加“自由生活”和“行动派”节奏。
- 增至 6 个犬种。
- 增加天气对事件权重的影响。
- 增加房间内可发现的小物件。
- 增加更多关系阶段和连续事件。

### 7.5 P2功能

- 装扮和生活章节付费。
- 小组件和锁屏状态。
- iCloud 多设备同步。
- 季节性故事包。
- 有限的自然语言对话。

## 8. 自由意志与叙事模型

### 8.1 状态模型

MVP状态仅用于内部决策，不以进度条展示：

- 心境：低落、平静、愉快、兴奋。
- 活力：休息、普通、活跃。
- 社交倾向：独处、中性、靠近用户。
- 好奇心：保守、中性、探索。
- 关系阶段：初识、熟悉、信任。

状态随时间和事件自然恢复，不依赖用户喂食维持。

### 8.2 事件选择

每个事件至少包含：

```text
event_id
time_window
required_traits
excluded_recent_events
required_memory_tags
state_effects
memory_output
follow_up_event_ids
scene_asset_id
authored_text_variants
```

事件评分建议：

```text
事件权重 = 基础权重
         + 性格匹配
         + 时间匹配
         + 记忆关联
         + 连续故事加成
         - 近期重复惩罚
```

### 8.3 AI使用边界

MVP默认使用人工编写的事件和多版本文案，不依赖云端大模型。

后续若引入 AI：

- 规则引擎先确定事件事实、人物、地点和结果。
- AI只能改写日记或对白语气。
- 输出必须经过结构化校验。
- 禁止 AI 自行创造健康危机、离家、死亡或责备用户的内容。
- 用户可关闭生成式内容并退回预设文案。

## 9. 用户故事与验收标准

### Epic假设

> 我们相信，为需要低负担数字陪伴的城市上班族提供一只拥有离线生活轨迹、关系记忆和有限自主性的数字狗，会让用户因为好奇与情感连接而主动回访。三天原型中，若至少 60% 的测试用户主动回来查看第二天内容，且至少 40% 完成全部三天体验，则该假设获得初步支持。

### Story 1：领养一只有差异的狗

作为首次用户，我希望快速了解并领养一只性格不同的狗，以便建立初始归属感。

**验收标准**

- 展示 3 只外观和性格不同的候选狗。
- 候选差异使用具体行为描述，而非抽象数值。
- 用户可以改名并确认领养。
- 不出现性别装扮限制或无限刷新。

### Story 2：回来发现离线生活

作为回访用户，我希望看到离开期间发生的事情，以便相信狗狗拥有连续生活。

**验收标准**

- 离开达到触发条件后生成 1-3 个事件。
- 首页明确区分当前状态与已经发生的片段。
- 事件带有时间和环境痕迹。
- 用户离开超过 24 小时不产生惩罚或损失。

### Story 3：回应狗狗

作为用户，我希望用很小的投入回应狗狗，并在未来看到回应留下的影响。

**验收标准**

- 每个重要片段提供 2-3 个回应和一个不回应选项。
- 选择在 500 毫秒内产生即时反应。
- 至少部分回应会在未来 24-48 小时被引用。
- 所有选择都不造成永久负面后果。

### Story 4：理解狗狗的自主选择

作为用户，我希望狗狗偶尔做出自己的决定，从而感觉它不是按钮驱动的玩偶。

**验收标准**

- 狗狗可以采纳、部分采纳或不采纳建议。
- 决定能够通过性格、状态或既有记忆解释。
- 同一建议不会永远产生同一结果。
- 自主性不表现为故意惩罚或对抗用户。

### Story 5：回顾共同经历

作为用户，我希望回看我们之间的重要片段，以便感受到关系在积累。

**验收标准**

- 时间线按日期显示事件和回应。
- 里程碑具有独立视觉标识。
- 用户可导出分享卡片。
- 用户可删除单条记录或全部数据。

## 10. 体验与视觉要求

### 10.1 视觉方向

- 原创温暖插画风，不使用“动物森友会风格”作为制作要求。
- MVP采用正面或轻微俯视的 2D 分层场景，不做 2.5D 等距开放场景。
- 首版只制作 1 个住所场景和 1 个街角外景。
- 狗狗素材采用分层结构，但每个犬种保留独特轮廓，不通过简单缩放替代。
- 动画优先级：呼吸、眨眼、走动、睡觉、回应。

### 10.2 交互要求

- 用户进入首页后 3 秒内理解狗狗正在做什么。
- 任何核心操作不超过 2 步。
- 不展示广告插屏。
- 不使用红点、倒计时、连续签到和损失厌恶文案。
- 支持减少动态效果和基础 VoiceOver 标签。

### 10.3 音频要求

- 默认音量温和，可一键静音。
- 使用原创或具备商业授权的环境音和短音效。
- MVP不制作连续背景音乐曲库。

## 11. 技术方案与数据

### 11.1 技术范围

| 项目 | MVP选择 |
|---|---|
| 平台 | iOS 17+ |
| UI | SwiftUI |
| 数据持久化 | SwiftData，本地优先 |
| 架构 | 轻量 MVVM 或单向数据流 |
| 事件引擎 | 本地确定性规则 + 带种子的加权随机 |
| 网络 | MVP非必需 |
| AI | MVP非必需 |
| 分析 | 隐私友好的基础事件埋点 |

### 11.2 核心数据实体

- `DogProfile`：犬种、外观、性格权重、领养时间。
- `DogState`：当前心境、活力、社交倾向、好奇心。
- `LifeEvent`：事件事实、发生时间、场景、状态影响。
- `Memory`：重要事件、用户回应、标签和引用关系。
- `Relationship`：关系阶段和里程碑，不使用连续签到。
- `NotificationPreference`：时间段、频率和总开关。

### 11.3 隐私

- MVP不采集精确位置、通讯录、健康数据或聊天内容。
- 用户可查看、导出和删除全部本地数据。
- 分析数据不得包含狗狗名称或用户自由输入文本。
- 后续引入云端 AI 前必须单独更新隐私说明和用户授权。

## 12. 成功指标与决策门槛

### 12.1 TestFlight验证阶段

样本：15-20 名目标用户，连续 3 天。

**主要指标**

- 第二天主动回访率 >= 60%。
- 三天完整体验率 >= 40%。

**行为指标**

- 至少 50% 的回访用户查看“你不在时”的全部新片段。
- 至少 40% 的用户在三天内产生一次可被后续引用的回应。
- 至少 30% 的用户主动询问后续故事、犬种或长期版本。

**质性指标**

- 至少 60% 的用户能用具体行为描述自己的狗狗性格。
- 至少 50% 的用户将回访理由表达为“想知道它发生了什么”或同义表述。
- 不超过 20% 的用户认为自己有照顾压力。

**付费信号**

- 至少 3 名测试用户愿意为新犬种、生活章节或完整版做真实预付/订金，而非仅口头表示。

### 12.2 Go / Pivot / Stop规则

- **Go：** 达成两项主要指标，且“好奇回访”质性指标成立。
- **Pivot：** 有回访但主要原因是装扮、喂养或提醒，而非狗狗生活连续性。
- **Stop候选：** 第二天主动回访低于 35%，或多数用户认为内容重复、狗狗像随机动画。
- 若触发 Stop 候选，须先排除崩溃、事件未生成、通知失效和首次体验不清楚等执行问题。
- 修复执行问题后最多再进行一轮同规模测试；若仍触发 Stop 条件，则停止当前方向。

### 12.3 后续产品阶段指标

只有原生 MVP 验证通过后才设置 7 日、30 日留存目标。首轮不预设 60% D7 或 40% D30，以实际测试数据建立基线。

### 12.4 工程验收门槛

- 冷启动后 3 秒内显示可理解的首页状态。
- 离线事件生成具备幂等性，同一时间段重复启动不得重复写入事件。
- 修改系统时间不得生成无限事件或破坏已有事件顺序。
- 带种子的事件选择可复现，便于测试事件因果关系。
- 数据模型升级不得删除领养记录和共同记忆。
- 核心领养、回访、回应和回顾流程完全离线可用。
- 分析或通知失败不得阻塞本地核心体验。
- Debug/TestFlight 构建提供时间推进、事件检查和状态检查入口，正式发布构建不包含这些入口。

## 13. 明确不做

验证版不包含：

- 20个犬种。
- 三种完整生活模式。
- 饥饿、口渴、生病、死亡和排泄系统。
- 现实任务、习惯打卡和运动同步。
- 无限 AI 对话。
- 散步小游戏和开放世界。
- 装扮商店、货币和抽卡。
- 用户社区、排行榜和共同养宠。
- 天气、季节和真实定位。
- 狗狗年龄和衰老。
- 面向用户的时间加速与冷却机制（开发和测试工具除外）。
- 周报、月报和复杂统计。

这些功能只有在核心回访假设成立、且能明确增强该假设时才重新评估。

## 14. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 用户认为狗狗只是随机动画 | 核心差异失败 | 增加因果事件、记忆引用和环境痕迹 |
| 三天后内容重复 | 留存快速下降 | 先设计可组合事件链，不盲目增加孤立文案 |
| 美术成本超过开发成本 | 无法按时验证 | 3只固定狗、2个场景、5个核心动画封顶 |
| 用户形成照顾负担 | 违背产品承诺 | 删除衰减、惩罚、红点和求救式通知 |
| AI生成内容不一致 | 破坏角色可信度 | MVP不用生成式AI，后续只允许受约束改写 |
| 竞品快速覆盖概念 | 差异缩小 | 重点积累原创角色、连续事件设计和关系记忆体验 |
| 用户不愿为陪伴付费 | 商业不可持续 | TestFlight 验证阶段进行真实预付测试，不以问卷意愿代替 |

## 15. 交付计划

### 阶段0：技术与内容设计，3-5天

- 定义数据模型、事件配置格式、状态变化和事件选择规则。
- 写出 M0 所需 15 个事件，并规划完整的 27 个事件。
- 完成核心页面流程和工程任务拆分。
- 评审事件因果、角色一致性和可测试性。

### 阶段1：内部可玩版本，2周

- 使用 SwiftUI 和 SwiftData 完成单犬、单场景版本。
- 打通领养、离线事件、回应、记忆引用和回顾闭环。
- 加入开发用时间推进与状态检查工具。
- 完成核心规则单元测试和内部体验检查。

### 阶段2：TestFlight MVP，2-4周

- 补齐 3 只候选狗、2 个场景和完整 P0 内容。
- 实现“我们的日子”、温柔通知、分享卡片和最小分析埋点。
- 完成 VoiceOver、减少动态效果、数据删除和基础稳定性检查。
- 分发 TestFlight 验证构建。

### 阶段3：三天封闭测试，5天

- 招募 15-20 名目标用户。
- 每天收集行为数据，结束后完成访谈和真实付费测试。
- 做 Go / Pivot / Stop 决策。

### 阶段4：验证后迭代

- Go 后才允许评估 P1、P2 和商业化系统。
- Pivot 时只调整被证据支持的核心循环，不同步扩大内容规模。
- Stop 时保留研究和技术资产，停止继续量产内容与美术。

## 16. 开放问题

1. 用户最想观察狗狗的生活，还是最想影响狗狗的生活？
2. 狗狗不采纳用户建议时，怎样表达才会显得有性格而非随机？
3. 片段更适合日记、漫画格、动态场景还是聊天式表达？
4. 用户是否愿意为犬种、生活章节或完整版本付费？
5. 应采用一次性买断、章节包还是订阅？验证前不做决定。
6. 狗狗是否需要明确职业，还是只表现城市生活节奏？
7. 关系记忆保留多久，怎样让用户理解它真正产生了影响？

## 17. 决策日志

| 决策 | 原因 |
|---|---|
| 从 iOS 15 提升至 iOS 17 | 与 SwiftData 技术选择一致，减少兼容实现 |
| MVP由20犬种缩减为3犬种 | 优先验证陪伴，不验证收集规模 |
| 删除用户性别选择 | 装扮偏好不应与性别绑定，且MVP无装扮商店 |
| 删除饥饿、口渴和连续签到 | 避免将陪伴转化为照料压力 |
| 删除年龄和衰老 | 尚未解决长期情感和生命周期问题 |
| 删除时间加速 | 真实等待是离线生活成立的一部分 |
| 暂不使用生成式AI | 先验证体验，避免成本和事实漂移 |
| 取消独立三天轻量原型 | 直接以原生 MVP 验证最危险的核心假设 |
| 跳过独立访谈与轻量原型 | 当前资源优先投入可运行产品，以原生 MVP 统一完成体验与需求验证 |
| 将 P0 拆为 M0 与 M1 | 尽早验证生活引擎闭环，避免等待全部美术与内容完成 |
| Stop 前允许一次执行问题修复 | 区分产品假设失败与原生版本质量问题，但限制重复试错成本 |

## 18. 下一步批准条件

本 PRD 批准进入原生 MVP 开发，但只批准 M0、M1 和 P0 验证范围。未经首轮 TestFlight 验证，不批准 P1、P2、20 犬种、美术量产、云端 AI 或商业化系统开发。

下一次评审必须携带：

- TestFlight 原生 MVP 录像和可访问构建。
- 三天封闭测试摘要和用户原话。
- 回访、完成和回应数据。
- 用户对狗狗性格与自主性的描述。
- 至少一次真实付费测试结果。

## 19. 用户体验完的一些反馈
1.用户体验的是无狗狗样式，狗狗也不能动的app，体验感比较差。整体界面应该要完善。
2.进入app，用户会发现提示，然后阅读片段，这里看到的都是文字。用户希望看到会动的画面，   真实的画面来展示用户不在的时候狗狗的动态。
3.用户没有看到狗狗的生活场景，生活场景很重要，是能够提升体验感的重要展现。也是后期商业化的重要手段。
