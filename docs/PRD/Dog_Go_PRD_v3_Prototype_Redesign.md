# Dog Go 产品需求文档（PRD）

**产品名称：** Dog Go  
**文档版本：** v3.0 原型重构版  
**文档状态：** 产品方案稿  
**目标平台：** iOS 17+，iPhone 竖屏  
**技术方向：** SwiftUI、SwiftData、本地优先、可测试的自主行为系统  
**核心主张：** 你的狗狗不等待指令，也不因你离开而受罚；它会在家里生活、探索、记住你们共同经历的日子。

---

# 1. 文档目的

本 PRD 基于 Dog Go 已形成的产品原型、角色视觉规范、五个核心生活场景、自主行为系统和跨场景生活闭环，对产品进行重新定义。

本版本重点解决以下问题：

1. 狗狗不能只是会眨眼、摇尾和切换姿态的动态图片；
2. 场景不能只是漂亮背景，而应成为行为、事件和记忆的载体；
3. 用户互动不能把狗狗变成遥控玩具；
4. 用户离开后，狗狗仍应拥有连续、可理解的生活；
5. 产品应传递陪伴感，而不是签到、喂养压力和惩罚机制；
6. 所有视觉、行为、事件和记忆应形成统一产品闭环。

---

# 2. 产品概述

Dog Go 是一款安静、治愈、低压力的虚拟宠物陪伴 App。

用户领养一只狗狗后，它会在一个温暖的家中自主生活。狗狗可以在客厅、玄关、卧室、餐厨和阳台之间移动，观察环境、休息、玩耍、等待用户、调查物品，并对用户的长期回应形成记忆。

用户不需要持续在线，也不会因离开、未签到或未完成任务而受到惩罚。

产品体验核心不是“照顾一只随时需要用户的宠物”，而是：

> 和一只有自己生活节奏的狗狗，共同建立长期关系。

---

# 3. 产品愿景

## 3.1 愿景

打造一个用户愿意每天短暂打开、静静看一会儿，并逐渐产生情感依恋的数字生命产品。

## 3.2 体验目标

用户使用 Dog Go 后，应形成以下感受：

- 它不是一个等待点击的功能入口；
- 它在我不在线时也会生活；
- 它能记住我对它的态度；
- 它不总是完全听我的；
- 它的行为有原因，不是随机动画；
- 它不向我索取注意力；
- 它让我感到陪伴，而不是负担。

## 3.3 一句话定位

> 一只有自己生活、会记住你、不会惩罚你的虚拟狗狗。

---

# 4. 产品原则

## 4.1 陪伴优先

所有设计首先服务于情绪陪伴，而不是留存指标、任务压力或成长数值。

## 4.2 狗狗拥有自主性

狗狗可以被用户影响，但不能像遥控器一样被完全控制。

## 4.3 行为必须有因果

每次主要行动应包含：

```text
感知
→ 需求变化
→ 形成意图
→ 执行动作
→ 得到结果
→ 更新状态
```

## 4.4 不惩罚离开

用户离开数小时或数天，不产生：

- 饥饿惩罚；
- 生病；
- 死亡；
- 遗弃指责；
- 连续签到中断；
- 资源损失；
- 求救文案。

## 4.5 画面优先，文字辅助

用户应主要通过场景和动作理解狗狗做了什么，文字只负责补充事实和情绪。

## 4.6 关系连续

用户的回应应留下影响，并在未来事件、行为偏好和日常回顾中被再次引用。

---

# 5. 目标用户

## 5.1 核心用户

- 喜欢狗狗，但现实条件不适合养狗的人；
- 希望获得安静陪伴的人；
- 喜欢治愈、低压力 App 的用户；
- 对虚拟宠物、数字生命、角色陪伴感兴趣的人；
- 希望每天进行 1–5 分钟轻量情绪休息的人。

## 5.2 非目标用户

首版不重点服务：

- 追求高强度游戏玩法的用户；
- 追求即时聊天陪伴的用户；
- 追求复杂数值养成的用户；
- 追求竞技、排行和社交炫耀的用户；
- 希望完全控制角色行为的用户。

---

# 6. 核心价值

## 6.1 真实陪伴

狗狗在家中拥有自己的生活节奏与活动空间。

## 6.2 环境互动

狗狗会感知窗帘、光斑、小鸟、纸袋、玩具、声音和用户归来。

## 6.3 自主行动

狗狗会基于状态、性格和环境自主选择行动。

## 6.4 情感连接

用户的回应会形成记忆，并影响后续行为与事件。

## 6.5 生活连续性

场景中的物品、痕迹和狗狗位置会延续之前发生的事情。

# 7. 产品信息架构

建议底部导航保留 4 个一级入口：

```text
家
日常
发现
我的
```

## 7.1 家

核心实时陪伴页面。

主要职责：

- 展示狗狗当前所在场景；
- 展示狗狗当前行为；
- 展示当前时间和环境；
- 提供低干扰互动；
- 显示未读生活片段；
- 允许跟随狗狗切换场景；
- 展示事件痕迹。

## 7.2 日常

记录狗狗已经发生的生活。

主要职责：

- 时间线展示生活片段；
- 展示事件画面；
- 展示用户回应；
- 展示后续记忆引用；
- 回顾重要关系节点。

## 7.3 发现

承载环境事件、探索内容和未来扩展。

首版职责：

- 展示可解锁的生活主题；
- 展示季节或特殊环境事件；
- 展示狗狗最近发现的物品；
- 不做强任务和签到。

## 7.4 我的

管理狗狗、设置和长期陪伴数据。

主要职责：

- 狗狗资料；
- 品种、名字和领养日期；
- 场景设置；
- 动态效果和无障碍设置；
- 隐私与本地数据；
- 陪伴数据；
- 关于 Dog Go。

# 8. 核心首页设计

## 8.1 首页结构

首页采用沉浸式场景作为主要视觉区域。

建议层级：

```text
顶部环境信息
→ 当前场景
→ 狗狗与环境
→ 当前状态轻提示
→ 低干扰互动入口
→ 底部导航
```

## 8.2 顶部信息

保留：

- 时间段；
- 天气或室内环境提示；
- 当前场景名称；
- 新生活片段数量。

不建议首版展示：

- 金币；
- 爱心数值；
- 连续签到；
- 经验值；
- 饥饿值。

这些元素容易让产品变成传统养成游戏，削弱真实陪伴感。

## 8.3 当前状态提示

状态文案应是观察性的，而不是命令性的。

推荐：

- “阳光很好，它在窗边待了一会儿。”
- “它刚从玄关回来。”
- “它似乎对纸袋很感兴趣。”
- “房间安静下来，它准备休息了。”

不推荐：

- “快来喂我！”
- “我已经等你很久了！”
- “今天还没有完成任务。”
- “亲密度即将下降。”

# 10. 场景连接关系

住宅关系：

```text
玄关
  ↓
客厅
  ├─ 阳台
  ├─ 餐厨
  └─ 卧室
```

## 10.1 场景切换原则

- 狗狗必须走向出口；
- 到达出口后才切换场景；
- 新场景从对应入口出现；
- 当前意图、朝向和动作序列保持连续；
- 不允许背景瞬间切换造成瞬移感；
- 用户可以跟随，也可以留在当前页面；
- 用户未跟随时，页面可以提示狗狗去了哪里。

# 11. 自主行为系统

## 11.1 自主行为结构

```text
DogState
→ DogPerception
→ DogNeeds
→ DogUtilityEvaluator
→ DogIntention
→ DogActionSequence
→ Animation / Scene Update
```

## 11.2 内部需求

首版采用：

```swift
struct DogNeeds {
    var rest: Double
    var curiosity: Double
    var play: Double
    var social: Double
    var comfort: Double
}
```

所有需求范围为 `0.0...1.0`。

## 11.3 主要目标

```swift
enum DogGoal {
    case rest
    case inspectWindow
    case inspectDoor
    case playWithToy
    case approachUser
    case investigateObject
    case relocateForComfort
    case remainQuietly
}
```

## 11.4 意图规则

- 一次只保留一个主意图；
- 意图具有最短持续时间；
- 不允许每几秒随机换目标；
- 高优先级环境刺激可打断低优先级行为；
- 用户互动可以影响狗狗，但不总是强制覆盖；
- 行为完成后才重新评估下一目标；
- 同一目标短时间内受到重复惩罚。

# 12. 核心行为链

## 12.1 窗帘变化

```text
窗帘移动
→ 转耳
→ 抬头
→ 形成好奇意图
→ 起身
→ 走到窗边
→ 坐下观察
→ 好奇需求下降
→ 决定停留或返回
```

## 12.2 用户离开

```text
用户离开
→ 狗狗从客厅走向玄关
→ 检查门口
→ 短暂等待
→ 无新刺激
→ 自主返回客厅
→ 选择休息位置
```

## 12.3 用户归来

```text
听见门锁或脚步
→ 转耳
→ 抬头
→ 判断当前行为是否可打断
→ 走向玄关或原地等待
→ 用户出现
→ 回头、靠近或轻摇尾
```

## 12.4 玩具调查

```text
发现玩具位置变化
→ 靠近
→ 闻或触碰
→ 玩耍
→ 玩具移动
→ 留下场景痕迹
```

# 13. 在线互动

首版提供三种轻互动：

## 13.1 叫名字

可能反馈：

- 立即回头；
- 只转耳；
- 延迟后靠近；
- 正在专注时暂不回应；
- 完成当前动作后再回应。

## 13.2 轻轻摸摸

可能反馈：

- 满足地眯眼；
- 调整姿势；
- 靠近一点；
- 轻微避开；
- 继续当前行为。

## 13.3 安静陪伴

可能反馈：

- 保持原位；
- 走到用户附近；
- 趴下；
- 减少活动；
- 一段时间后更换舒适位置。

## 13.4 互动原则

- 不产生奖励货币；
- 不产生惩罚；
- 不强制打断狗狗；
- 反馈应符合当前状态；
- 重要互动可以写入记忆。

# 14. 日常与记忆

## 14.1 生活片段

用户离开后，狗狗可以发生 1–3 个生活片段。

每个片段包含：

- 事件事实；
- 场景；
- 狗狗行为；
- 情绪；
- 可见痕迹；
- 用户回应；
- 后续影响。

## 14.2 我们的日子

用于展示：

- 领养日；
- 首次分别；
- 首次主动靠近；
- 首次记住用户回应；
- 特殊事件；
- 场景变化；
- 被未来事件引用的记忆。

## 14.3 记忆引用

后续事件可以引用：

- 用户曾经安静陪伴；
- 用户曾鼓励它去观察；
- 用户曾选择不过度打扰；
- 狗狗曾把玩具藏起来；
- 狗狗曾在玄关等待。

---

# 15. 发现模块

发现不应变成任务中心。

首版可包含：

- 最近发现；
- 生活主题；
- 季节变化；
- 场景中的新物品；
- 特殊自然事件；
- 连续生活故事入口。

不建议：

- 每日任务；
- 强制签到；
- 倒计时活动；
- 高压限时奖励；
- 竞技排行。

---

# 16. 装扮与道具

原型中可保留轻量装扮方向，但建议放在 M1.5 或更晚。

可装扮：

- 围巾；
- 狗窝；
- 小毯子；
- 水碗；
- 玩具；
- 场景小摆件。

设计原则：

- 不影响核心行为；
- 不改变狗狗性格；
- 不通过高压付费获得；
- 不造成视觉风格破坏；
- 所有装扮需支持场景和动画。

首版不建议加入货币系统。

# 17. 状态系统

## 17.1 长期状态

建议保存：

- 名字；
- 品种；
- 性格权重；
- 心境；
- 精力；
- 社交倾向；
- 好奇倾向；
- 最近所在场景；
- 最近所在锚点；
- 重要记忆；
- 场景痕迹；
- 关系阶段。

## 17.2 短期状态

仅保存在运行时：

- 当前意图；
- 当前行动步骤；
- 当前路径；
- 当前动画 token；
- 临时刺激；
- 临时文案；
- 动作进度。

## 17.3 不建议直接显示的数值

内部可使用，但不应全部展示：

- 精力；
- 社交；
- 好奇；
- 舒适；
- 玩耍需求。

用户看到的应是可理解的状态描述，而不是仪表盘。

# 18. 视觉与动画规范

## 18.1 角色风格

```text
温暖轻绘本
+ 轻拟人比例
+ 简化毛发
+ 高表情可读性
+ 清晰轮廓
+ 动画友好
```

## 18.2 场景风格

```text
柔和低对比
+ 简化空间细节
+ 大块自然光影
+ 纸张纹理
+ 角色视觉优先
```

## 18.3 动画层级

```text
宏观行动
→ 起身、转向、移动、到达

身体动作
→ 坐、趴、站、玩耍、睡眠

微动作
→ 呼吸、眨眼、转耳、尾巴、表情
```

## 18.4 动画原则

- 动作必须服从意图；
- 微动作不能随机乱播；
- 移动不能只做图片平移；
- 同一关节不能被多个通道冲突驱动；
- 行为必须有开始、过程和结束；
- 减少动态效果开启后保留逻辑，弱化表现。

# 23. 核心验收标准

## 23.1 产品验收

- 用户可以完成领养和命名；
- 狗狗在用户离开后发生生活事件；
- 用户可以查看和回应；
- 回应可以被后续事件引用；
- 用户离开超过 24 小时不受惩罚；
- 首页可以观察狗狗自主生活；
- 狗狗不依赖点击也能行动；
- 用户能够理解狗狗为什么移动。

## 23.2 自主行为验收

- 狗狗至少有 4 类自主目标；
- 当前意图具有稳定性；
- 行为具有开始、过程和结束；
- 同一行为不会机械重复；
- 用户互动不会总是强制覆盖当前意图；
- 至少完成一条跨场景行为链；
- 场景切换无瞬移感。

## 23.3 视觉验收

- 狗狗是虚拟角色，不是写实宠物插画；
- 场景风格统一；
- 角色始终是视觉中心；
- 场景痕迹可独立开关；
- 日夜变化自然；
- 小屏下表情清晰；
- 动画不呈现贴纸滑动感。

## 23.4 性能与无障碍

- 常用 iPhone 尺寸运行流畅；
- 无明显持续高耗电；
- 支持减少动态效果；
- 支持 VoiceOver；
- 支持动态字体；
- 核心互动在 500 ms 内出现反馈。

# 25. 关键风险

## 25.1 狗狗仍像动态图片

控制方式：

- 引入空间移动；
- 引入意图；
- 引入行动序列；
- 减少随机微动作。

## 25.2 场景过度复杂

控制方式：

- 使用固定锚点；
- 先开发两个场景；
- 不做自由寻路；
- 分层资产标准化。

## 25.3 美术一致性不足

控制方式：

- 冻结角色母版；
- 冻结场景光照；
- 统一命名与锚点；
- 所有资产先通过真机视觉验收。

## 25.4 产品变成传统养成游戏

控制方式：

- 不引入饥饿惩罚；
- 不引入签到；
- 不引入强货币系统；
- 关系感优先于数值。

## 25.5 自主行为不可控

控制方式：

- 使用 Utility AI；
- 当前意图保持；
- 行为冷却；
- 重复惩罚；
- 稳定随机源；
- 单元测试。

# 27. 一句话产品定义

> Dog Go 不是一个需要用户不断照顾的虚拟宠物游戏，而是一只会在家中自主生活、记住共同经历，并在用户回来时继续陪伴的数字狗狗。

# 28. 最终结论

本产品的核心竞争力不在于动画数量、场景数量或数值系统，而在于：

```text
自主行为
+ 空间生活
+ 环境痕迹
+ 关系记忆
+ 无压力陪伴
```

只要用户能够在画面中理解：

> “它刚才听见了门外的声音，所以自己去玄关看了一会儿，然后又回到客厅休息。”

Dog Go 就完成了从虚拟宠物工具向数字生命陪伴产品的关键升级。
